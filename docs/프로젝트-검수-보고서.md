## 1. 잘 되어 있는 점 (유지할 강점)

- **보안 위생 우수**
  - `.gitignore`가 `.env`·`backend/.env`·`.env.local`을 정확히 차단. **실제 시크릿이 git 히스토리에 커밋된 적 없음**(검증 완료).
  - 추적되는 `frontend/.env.production`은 플레이스홀더(`api.example.com`), `application-prod.yml`에 하드코딩 시크릿 없음.
  - `SecurityConfig`: CSRF 비활성(JWT+SameSite 쿠키 정책상 적절), formLogin/httpBasic 차단, `permitAll` 범위가 토큰 재발급·OAuth·Swagger·공개 이미지 프록시로 한정. 나머지 `authenticated`. 커스텀 401/403 JSON 핸들러.
  - 정산 계좌·링크 **암호화 저장**(`PayoutCipher`/`PayoutCryptoConverter`, 마이그레이션 `V27`/`V29`).
- **DB 초기화 이중 안전장치**: `FlywayResetConfig`가 `app.db-reset-on-start` 플래그 + `prod` 프로필 강제 차단으로 운영 데이터 보호.
- **백엔드 테스트 충실**: accommodation·auth·expense·group·home·notification·place·recommend·schedule·settlement 등 도메인별 서비스 테스트 보유.
- **데이터 무결성/성능 기초**: 마이그레이션에 인덱스/유니크 제약 **30건**, Bean Validation **33개 파일**, JPA 연관관계 **EAGER 0건**(기본 LAZY).
- **문서화 풍부**: `docs/`에 요구사항(`requirements.md`)·배포 가이드·로컬 개발 가이드·핸드오프 문서 등.
- **TODO 부채 거의 없음**: 전체 코드에서 TODO/FIXME 단 2건.

## 2. 발견 사항 (영역별 · 심각도순)

### 🔴 2-1. CI/CD 파이프라인 부재
- `.github/workflows` 없음. push/PR 시 **자동 빌드·테스트·린트가 돌지 않음**.
- 백엔드 테스트가 잘 갖춰져 있는데 **자동 실행되지 않으면 회귀를 놓치기 쉬움**.
- **권장**: GitHub Actions로 (1) 백엔드 `./gradlew test` (2) 프론트 `tsc -b && vite build && eslint` (3) main/develop 보호 규칙. 컨테이너 빌드까지 확장 가능.

### 🔴 2-2. 프론트엔드 테스트·품질 도구 전무
- 테스트 프레임워크(Vitest/Testing Library) 없음 — 프론트 자동 테스트 **0건**.
- Prettier/husky/lint-staged 없음 — 포맷 일관성·커밋 전 검사 미적용. (CRLF/LF 혼재 경고도 관측됨)
- **권장**: Vitest + React Testing Library 도입(핵심 플로우: 로그인·그룹생성·정산부터). Prettier + `.editorconfig` + (선택) husky pre-commit으로 `tsc`·`eslint`·`prettier --check`.

### 🟡 2-3. 성능 — 번들 코드스플리팅 부재
- `React.lazy`/`Suspense` 사용 **0건** → JS 단일 번들 **약 552KB(gzip 164KB)** 를 첫 진입에 통째로 로드. 모바일 초기 로딩 불리.
- **권장**: 라우트 단위 `React.lazy` + `Suspense`로 분할. 특히 대형 페이지(아래 2-6)와 `recharts`(차트) 지연 로드.

### 🟡 2-4. 확장성 — 목록 API 페이지네이션 부재
- `Pageable`/`Page<>` 사용 **0건**. 보관함·지출 등 목록을 전량 반환.
- 친구 단위 데이터량(그룹당 수십 건)에선 당장 문제 없음. 알림은 SSE 스토어 기반이라 서버 페이지네이션 대상 아님.
- **결정(2026-06-26)**: 렌더 비용은 **클라이언트 "더보기"**(보관함 기존 + 지출 신규 적용, PAGE_SIZE=8)로 해소. 데이터 규모상 백엔드 `Pageable` 도입은 과설계로 판단해 **보류**.
- **향후(데이터 증가 시)**: 보관함·지출 조회를 Spring Data `Pageable`로 전환 + 프론트 `useInfiniteQuery` 무한스크롤.

### 🟡 2-5. 데이터 패칭 패턴 불일치
- `@tanstack/react-query` 설치·**11개 파일에서 사용**하지만, **21개 페이지는 raw `useEffect`+axios** 로 수동 패칭(캐시·중복요청·로딩/에러 처리 제각각). 예: `HomePage`, `MyPage`.
- **권장**: react-query로 점진 통일(쿼리키 컨벤션은 이미 `src/queryKeys` 존재). 수동 패칭은 신규/리팩토링 시 우선 전환.

### 🟡 2-6. 대형 컴포넌트/서비스 (유지보수성)
- 프론트: `TripPlanPage.tsx` **992줄**, `ScheduleBuilderPage` 644, `ScheduleMapPage` 612, `MyPage` 532.
- 백엔드: `ExpenseService` 461, `GroupService` 404.
- **진행(2026-06-26)**: `MyPage`(532→302) 분해 — 정산(`PayoutSection`)·통계+배지(`TripStatsSection`)를 자체 조회를 가진 독립 섹션으로 분리.
- **후속 과제(앱 실행 검증 필요)**: `TripPlanPage`는 30여 개 상호의존 상태를 가진 다단계 마법사라, 회귀 방지를 위해 **로컬에서 플로우를 직접 돌리며** 스텝 단위 컴포넌트/훅으로 분해 권장. `ScheduleBuilderPage`·`ScheduleMapPage`도 동일.

### 🟡 2-7. 기술 스택이 전부 bleeding-edge
- Spring Boot **4.0.6**, Java 21, React **19.2**, Vite **8**, Tailwind **4**, TypeScript **6**, react-router **7**.
- 모두 최신/극초기 버전 → 라이브러리 호환성·레퍼런스 부족·미성숙 이슈 가능성. (실제 `SecurityConfig` 주석에 Spring Boot 4의 `-parameters` 관련 우회 흔적 존재)
- **권장**: 버전 고정(lockfile 커밋 확인) + 업그레이드 시 회귀 테스트 의존. 신규 도입 라이브러리는 해당 메이저 호환성 사전 확인.

### 🟢 2-8. Swagger 운영 노출 점검
- `springdoc` 활성화 기본값이 `SWAGGER_ENABLED:true`, SecurityConfig에서 `permitAll`. 운영에선 `SWAGGER_ENABLED=false`로 확실히 끄는지 배포 체크리스트에 명시 필요.

### 🟢 2-9. 잔여 housekeeping
- 루트 `enjoytriprename.patch`(28KB) — 리네임 작업 잔재, 삭제 권장.
- `편돌즈_디스코드_로고.png`(1.2MB) 미추적 — 에셋 위치 정리(예: `docs/` 또는 `frontend/public`)·LFS 여부 결정.
- 백엔드 내부 `spring.application.name: enjoy-trip` 잔존(사용자 비노출이라 무해하나 일관성 차원에서 정리 가능).
- `favicon.svg`가 아직 옛 enjoy-trip 퍼플 마크 — 새 로고 기반 교체 필요(브랜딩 미완 항목).

## 3. 다음 개발 방향성 (로드맵)

### 단기 (1~2주 · 실사용 직전 안정화)
1. **배포 마무리**: `frontend/.env.production`의 실제 API 도메인, `FRONTEND_BASE_URL`/`CORS_ALLOWED_ORIGINS`, **카카오·구글 콘솔 Redirect URI를 새 도메인으로 갱신**, prod에서 Swagger off·`DB_RESET_ON_START=false` 확인.
2. **CI 최소 구성**: GitHub Actions에서 백엔드 테스트 + 프론트 빌드/린트 자동화.
3. **브랜딩 마감**: 새 favicon/앱아이콘(디스코드 로고 기반), OG 메타.
4. **에러 모니터링 도입**: Sentry(프론트+백엔드) — 실사용 시 장애 가시화.

### 중기 (한 달 내 · 신뢰성·UX)
5. **프론트 테스트 베드**: Vitest+RTL로 로그인·그룹생성·정산·투표 핵심 플로우 커버.
6. **코드스플리팅 + 페이지네이션**으로 성능/확장성 확보.
7. **데이터 패칭 react-query 통일** 진행.
8. **정산 기능 심화**(서비스 핵심 가치): 정산 내역 히스토리, 미정산 리마인드 알림, 분할 규칙(N분의 1/비율/지정), 송금 완료 체크.

### 장기 (성숙도)
9. **접근성 점검**(alt/aria는 일부 양호 — 전반 audit), 다크모드 잔여 하드코딩 색 마감.
